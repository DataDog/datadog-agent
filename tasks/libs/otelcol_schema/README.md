# DDOT Collector configuration JSON Schema bundle

Tooling that produces a single JSON Schema 2020-12 document validating a
complete OpenTelemetry Collector YAML config as accepted by the Datadog
Distribution of OpenTelemetry (DDOT) Collector. The output is checked in at
`comp/otelcol/collector-config.schema.json` and consumed by Fleet Automation
for remote-config validation, IDE schema stores, and documentation
generation.

## What ships

| Path | Purpose |
|---|---|
| `comp/otelcol/collector-config.schema.json` | The artifact. Regenerated by this tooling; marked `linguist-generated=true`. |
| `tasks/libs/otelcol_schema/inventory.py` | Audits which manifest components have a published `config.schema.yaml`. |
| `tasks/libs/otelcol_schema/convert.py` | Converts a single `config.schema.yaml` to a JSON Schema fragment. |
| `tasks/libs/otelcol_schema/bundle.py` | The bundler. Builds the registry, stitches the envelope, emits the bundle. |

| `tasks/otelcol_schema.py` | Invoke task wrappers (`dda inv otelcol-schema.*`). |
| `tasks/unit_tests/otelcol_schema_tests.py` | 98 unit tests pinning the surface. |

## How to rebuild

```bash
dda inv otelcol-schema.gen
```

That's the whole story for the happy path. The task:

1. Pre-flights `go mod download` for any manifest-pinned module missing from
   the Go module cache (fresh-clone safe).
2. Walks the OCB manifest at `comp/otelcol/collector-contrib/impl/manifest.yaml`
   for component schemas.
3. Consults `UPSTREAM_SHIMS` in `bundle.py` for in-repo overrides of
   in-flight upstream PRs.
4. Reads local Datadog-authored schemas under `comp/otelcol/**`.
5. Walks transitive `$ref`s into shared types in the Go module cache.
6. Stitches the hand-authored envelope on top of the `$defs` registry.
7. Writes the result to `comp/otelcol/collector-config.schema.json`.

Verify the artifact is up-to-date:

```bash
dda inv otelcol-schema.check
```

This regenerates into a tempfile and diffs against the committed artifact;
non-zero exit on drift. Suitable as a CI gate.

Audit which components have a published schema:

```bash
dda inv otelcol-schema.inventory --json /tmp/i.json --md /tmp/i.md
```

## Common maintenance tasks

### Manifest version bumped (new Collector release)

When `comp/otelcol/collector-contrib/impl/manifest.yaml`'s `dist.version`
or any `gomod` pin changes:

1. Run `dda inv tidy` to refresh `go.mod` / `go.sum`.
2. Run `dda inv otelcol-schema.gen` to regenerate the bundle. The
   `x-collector-version` field in the bundle output auto-tracks
   `dist.version`.
3. Run `dda inv otelcol-schema.check` to verify nothing else drifted.
4. Decide whether the new Collector release ships any schemas that retire
   one of our shims (see below).
5. Don't bump `x-bundle-version` for this — Collector version bumps are
   patch-level for the bundle format.

### Datadog-authored component config struct changed

Any change to a `Config` struct under `comp/otelcol/**` (e.g. adding a
field, renaming, changing types) needs the component's `config.schema.yaml`
regenerated by upstream's `schemagen`. The repo-level `.schemagen.yaml` and
per-component `metadata.yaml` files are already in place, so:

```bash
# From the component's directory
(cd ~/path/to/opentelemetry-collector-contrib/cmd/schemagen && go build -o /tmp/schemagen .)
/tmp/schemagen comp/otelcol/<...>/path/to/component
```

The bundler picks up the refreshed local `config.schema.yaml` on next
`otelcol-schema.gen` run.

### A new Datadog-authored component is added

1. Generate its `config.schema.yaml` (as above).
2. Add the component to `LOCAL_SCHEMAS` in `inventory.py`.
3. Run `dda inv otelcol-schema.gen`.
4. Confirm the new component appears in the bundle's
   `properties.<class>.patternProperties` and as a `component__<class>__<type>`
   entry under `$defs`.

### Upstream PR #15300 merges (or a new Collector release picks it up)

Retire the shim layer. Per
`tasks/libs/otelcol_schema/upstream_shims/README.md`:

1. Bump the manifest's pinned Collector version to one that includes the
   upstream schemas.
2. Run `dda inv otelcol-schema.gen` — the regenerated bundle now sources
   from the real cached upstream module.
3. Run `dda inv otelcol-schema.check` and confirm the diff against the
   pre-deletion committed artifact is zero. Byte-identical output means the
   shim was a faithful preview of what upstream merged.
4. Delete the relevant entries from `UPSTREAM_SHIMS` in `bundle.py` and the
   subdirectories under `upstream_shims/`.
5. Run `dda inv otelcol-schema.gen` once more to refresh the committed
   artifact (no schema diff — only the in-repo file moves).

### A new upstream-PR shim needs to be added

1. Create `tasks/libs/otelcol_schema/upstream_shims/<flat_path>/` with the
   schema file and `metadata.yaml` from the upstream PR.
2. Add an entry to `UPSTREAM_SHIMS` in `bundle.py` mapping the upstream
   Go import path to the new shim directory.
3. Update `upstream_shims/README.md` with the new entry.
4. Run `dda inv otelcol-schema.gen` and add a test in
   `TestUpstreamShims` that pins the new entry.

### The bundle format changes (validators must react)

Bump `BUNDLE_SCHEMA_VERSION` in `bundle.py`:

- **MAJOR:** envelope shape, `$defs` naming scheme, or `$ref` form changes
  in ways that break existing validators (e.g. renaming
  `component__<class>__<type>` to a different scheme).
- **MINOR:** new validator-visible affordances (new `x-*` extension fields,
  new placeholder shapes, new permissive-fallback semantics).
- **PATCH:** schema content updates that don't change format (the most
  common case — Collector version bumps, shim adds/retires, upstream
  schema refreshes).

After bumping, run `dda inv otelcol-schema.gen` to refresh the artifact and
ensure tests still pass.

## Inputs the bundler reads

The bundler consults three sources for per-component schemas, in this order
(first match wins):

1. **`UPSTREAM_SHIMS`** in `bundle.py` — in-repo `config.schema.yaml` files
   under `upstream_shims/<flat_path>/`. Temporary; tied to in-flight
   upstream PRs.
2. **`LOCAL_SCHEMAS`** in `inventory.py` — Datadog-authored components
   under `comp/otelcol/**`. These are the source-of-truth for our own
   components.
3. **Go module cache** — `~/go/pkg/mod/...` populated by `go mod download`.
   Hosts both upstream contrib schemas and upstream core (Collector
   library) schemas.

For shared types referenced via `$ref`, the bundler follows the chain
transitively, consulting the same three sources for each target.

## Bundle output structure

```jsonc
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id":     "https://github.com/DataDog/datadog-agent/comp/otelcol/collector-config.schema.json",
  "title":   "Datadog Distribution of OpenTelemetry Collector configuration",
  "x-bundle-version":    "1.0.0",       // semver of the bundle FORMAT
  "x-collector-version": "v0.151.0",    // upstream Collector version
  "description": "...",
  "type":     "object",
  "additionalProperties": false,
  "properties": {
    // top-level envelope: receivers / processors / exporters / connectors /
    // extensions / service. Each component-class block has patternProperties
    // keyed on `^<type>(/.*)?$` pointing into $defs.
  },
  "$defs": {
    // ~300 entries: component schemas + shared types + missing-component
    // placeholders.
  }
}
```

See the top of `bundle.py` for the `$defs` naming scheme.

## Testing

```bash
python -m unittest tasks.unit_tests.otelcol_schema_tests
```

98 tests pin the public surface: ref classification, ID generation, the
full `_canonicalise_ref` dispatch table, envelope stitching, the
`gen`/`check`/`inventory` invoke tasks, the upstream-shim mechanism, and
bundle versioning. Tests that need filesystem fixtures use `tempfile`;
they don't touch the real Go module cache.

## CI integration

Not wired today. The natural follow-up is a GitLab linter-stage step that
runs `dda inv otelcol-schema.check` and fails if the committed artifact
has drifted from the regenerated output. Track in the plan doc when
adding.
