# DDOT collector

-----

DDOT (Datadog Distribution of the OpenTelemetry Collector) is a full, user-configurable OpenTelemetry Collector shipped as its own binary, `otel-agent` ([`cmd/otel-agent`](<<<SRC>>>/cmd/otel-agent)). It is built from a curated set of upstream collector-contrib components plus Datadog-specific ones, and â€” unlike the OSS Datadog exporter, which talks to the intake with its own HTTP clients â€” it exports through the Agent's classic pipelines: metrics through `pkg/serializer` and the [forwarder](../pipelines/forwarder.md), logs through a logs agent running in-process, and traces through an in-process [trace-agent](../pipelines/traces.md) component. DDOT normally runs *connected* to a core Agent on the same host or pod (remote tagger, hostname, config sync, and secret resolution over authenticated IPC), and also supports standalone and gateway modes. For the small hardcoded collector embedded in the core agent, see [OTLP ingest](otlp-ingest.md).

## The binary and its configuration inputs

`otel-agent run` takes two families of configuration:

1. `--config` â€” one or more OTel collector config URIs, exactly as the upstream collector accepts them. Supported schemes are `file:`, `env:`, `yaml:`, `http:`, and `https:`, with `env` as the default scheme. `--set key=value` overrides are translated into `yaml:` URIs using `::` as the key delimiter.
1. `--core-config` (env `DD_CORE_CONFIG`) â€” a `datadog.yaml` used for Agent-side settings (IPC, logging, `otelcollector.*`).

If `DD_FLEET_POLICIES_DIR` is set and contains an `otel-config.yaml`, `buildConfigURIs` in [`run/command.go`](<<<SRC>>>/cmd/otel-agent/subcommands/run/command.go) appends it after the user files â€” this is how [Fleet Automation](../deployment/fleet.md) layers remotely managed collector config on top of local config. Effective precedence, lowest to highest: `--config` files in order, then the fleet policy file, then `--set` overrides.

The process refuses to start unless `otelcollector.enabled: true` (`DD_OTELCOLLECTOR_ENABLED`) is set in the core config â€” the Helm chart and Operator set this when the DDOT container is enabled.

The default config shipped with the binary ([`cmd/otel-agent/dist/otel-config.yaml`](<<<SRC>>>/cmd/otel-agent/dist/otel-config.yaml)) wires an `otlp` receiver on `0.0.0.0:4317`/`4318`, a `prometheus` receiver scraping the collector's own telemetry on `:8888`, the `infraattributes` processor at high cardinality (`cardinality: 2`), the `datadog` exporter with `${env:DD_API_KEY}`, and the `datadog/connector` so that APM stats are computed for traces.

### Config bridging: building a Datadog config from OTel config

DDOT inverts the usual direction of configuration: `NewConfigComponent` in [`cmd/otel-agent/config/agent_config.go`](<<<SRC>>>/cmd/otel-agent/config/agent_config.go) resolves the collector config first, locates the (single) `datadog` exporter in it, and *copies its settings into the Datadog config object* that all the embedded Agent components consume. Copied fields include `api.key`, `api.site`, the metrics endpoint (as `dd_url`), logs endpoint and compression, proxy settings, and trace options (`ignore_resources`, `trace_buffer`, `apm_config.features` derived from collector feature gates). The effective log level is the more verbose of the Agent `log_level` and the collector's `service::telemetry::logs::level`.

Several settings are forced during bridging, and they define DDOT's relationship to the classic trace pipeline:

1. `apm_config.receiver_enabled: false` â€” the embedded trace-agent opens **no** TCP receiver; port 8126 does not exist in DDOT. Spans enter only via the in-process `ReceiveOTLPSpans` API.
1. `otlp_config.traces.internal_port: 0` â€” the [OTLP ingest](otlp-ingest.md) loopback hop is disabled; there is no embedded-collector-to-trace-agent port inside DDOT.
1. `apm_config.apm_non_local_traffic: true`, and the metrics serializer is pinned to zlib compression with the v3 series API disabled (the v3 intake requires a compression scheme DDOT does not use â€” this is asserted in both `agent_config.go` and the serializer exporter).

If the collector config contains no `datadog` exporter at all (`ErrNoDDExporter`), the binary still runs â€” but as a bare collector with converter, remote tagger, and IPC only, and none of the Agent export pipelines. If it contains more than one `datadog` exporter, bridging fails with `multiple datadog exporters found`.

## Fx wiring: connected, standalone, and gateway

`runOTelAgentCommand` ([`run/command.go`](<<<SRC>>>/cmd/otel-agent/subcommands/run/command.go)) assembles the component graph. The always-present core: the default [forwarder](../pipelines/forwarder.md), the shared serializer (zlib), a logs-agent pipeline ([`comp/otelcol/logsagentpipeline`](<<<SRC>>>/comp/otelcol/logsagentpipeline), intake origin `DDOTIntakeOrigin`), the in-process trace-agent component ([`comp/trace/agent`](<<<SRC>>>/comp/trace/agent)), [workloadmeta](../containers/workloadmeta.md), a [tagger](../containers/tagger.md), the converter, the collector-contrib factories, and the collector component itself. On top of that, one of two mutually exclusive option sets:

1. **Connected mode** (default): tag lookups are proxied to the core agent's tagger over gRPC (remote tagger; disabled when `cmd_port <= 0`), the hostname comes from the core agent over [IPC](../processes/ipc.md) with a local fallback, and `configsync` pulls the core agent's resolved configuration from its `/config/v1` IPC endpoint at `agent_ipc.host:agent_ipc.port` (with periodic refresh per `agent_ipc.config_refresh_interval`). Secrets are a no-op resolver: `ENC[...]` handles are resolved by the core agent and arrive via config sync â€” see [Secrets management](../configuration/secrets.md).
1. **Standalone mode** (`otel_standalone: true` / `DD_OTEL_STANDALONE=true`): a local workloadmeta-backed tagger, local hostname resolution, a real in-process secrets resolver, and host-metadata collection modules for the `dogtel` extension. Standalone forces `cmd_port: -1` (no core-agent IPC at all) and disables remote configuration.

Gateway deployments (`otelcollector.gateway.mode: true`, set by Helm/Operator env `DD_OTELCOLLECTOR_GATEWAY_MODE`) are standalone-like: there is no core agent next to the collector, so the remote tagger and IPC are off, and the `datadog` extension reports `deployment_type: gateway`. Gateway usage telemetry is tracked by [`pkg/util/otel/gatewayusage.go`](<<<SRC>>>/pkg/util/otel/gatewayusage.go) and is also inferred from resource attributes when spans traverse a daemonset collector *and* a gateway.

/// note
There is a deliberate Fx ordering hack: trace-config construction depends on `configsync.Component` so that the API key has been synced from the core agent before the embedded trace-agent starts (noted in `run/command.go`).
///

The collector component ([`collector/impl/collector.go`](<<<SRC>>>/comp/otelcol/collector/impl/collector.go)) merges the contrib factories with the Datadog ones in `addFactories` â€” the `datadog` exporter, `infraattributes` processor, `datadog` connector, and the `ddflare`, `ddprofiling`, and `dogtel` extensions â€” then validates the effective config with `DryRun` before calling `Run`.

## The converter

[`comp/otelcol/converter/impl`](<<<SRC>>>/comp/otelcol/converter/impl) implements `confmap.Converter` and rewrites the resolved collector config *before* the collector starts. It is enabled by `otelcollector.converter.enabled` (default `true`), with the feature list `otelcollector.converter.features` defaulting to `infraattributes, prometheus, pprof, zpages, health_check, ddflare, datadog`. For each missing piece it injects a component named `<name>/dd-autoconfigured`:

1. Extensions ([`extensions.go`](<<<SRC>>>/comp/otelcol/converter/impl/extensions.go)): `pprof` (:1777), `zpages` (:55679), `health_check` (:13133), `ddflare`, and the upstream `datadog` extension, which feeds the Fleet Automation inventory (API key, site, `deployment_type` daemonset or gateway, hostname, `otelcollector.installation_method`). If the user declared a `datadog` (or `dogtel`) extension but forgot to list it under `service::extensions`, the converter wires the existing one in rather than duplicating it.
1. The `infraattributes` processor ([`infraattributes.go`](<<<SRC>>>/comp/otelcol/converter/impl/infraattributes.go)) is inserted into every pipeline that ends in a `datadog` exporter.
1. A `prometheus` receiver ([`prometheus.go`](<<<SRC>>>/comp/otelcol/converter/impl/prometheus.go)) scraping the collector's internal telemetry endpoint (`service::telemetry::metrics`, default `:8888`) is wired to each `datadog` exporter not already covered by a user-configured scrape, through a dedicated `metrics/dd-autoconfigured/<exporter>` pipeline with a filter processor that drops `scrape_*` noise â€” this is how `otelcol_*` self-health metrics reach Datadog.
1. `addCoreAgentConfig` ([`coreconfig.go`](<<<SRC>>>/comp/otelcol/converter/impl/coreconfig.go)) copies `api_key` and `site` from the core Agent config into any `datadog` exporter that lacks them, and replaces an unresolved `ENC[...]` API key with the core agent's resolved value.
1. In standalone mode, a `dogtel/dd-autoconfigured` extension is added.
1. The converter warns (but does not block) when a `hostmetrics` receiver is configured in connected mode, since it duplicates the core agent's system metrics.

Because the converter mutates configuration invisibly, the [`ddflare` extension](<<<SRC>>>/comp/otelcol/ddflareextension/impl) keeps three views: the **provided** config (pre-converter, with env-var references re-substituted), the **enhanced** config (post-converter, what actually runs), and the env-var table. All three are served as JSON over its HTTPS endpoint for flares and Fleet Automation.

## The datadog exporter: mapping onto classic pipelines

[`datadogexporter/factory.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/datadogexporter/factory.go) registers traces, metrics, and logs exporters under the type `datadog`, using the *upstream* config schema (`open-telemetry/opentelemetry-collector-contrib/pkg/datadog/config`) so user YAML written for the OSS exporter is drop-in compatible. What differs is where the data goes:

```text
                          +----------------- otel-agent process ------------------+
                          |                                                        |
  OTLP / prometheus /     |  collector pipelines                                   |
  filelog / ... --------->|    traces  --> datadog exporter --ReceiveOTLPSpans--> in-process trace-agent --> APM intake
                          |       \--> datadog connector (APM stats as OTLP metrics)
                          |    metrics --> datadog exporter --> serializerexporter --> pkg/serializer --> forwarder --> metrics intake
                          |                          \-- stats channel --> trace-agent SendStatsPayload
                          |    logs    --> datadog exporter --> logsagentexporter --> in-process logs agent --> logs intake
                          +--------------------------------------------------------+
```

1. **Traces** ([`traces_exporter.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/datadogexporter/traces_exporter.go)): each `ResourceSpans` batch is handed straight to the trace-agent component's `ReceiveOTLPSpans` with the header `Datadog-Client-Computed-Stats: true`, so the trace-agent's concentrator skips stats computation for these spans (stats come from the `datadog` connector instead). The exporter deliberately disables retry (to avoid APM event duplication) and the exporterhelper timeout. The embedded trace-agent then obfuscates, samples, and forwards per `apm_config.*` â€” the same pipeline described in [Trace pipeline (APM)](../pipelines/traces.md), minus the HTTP receiver.
1. **Metrics**: the exporter wraps a [`serializerexporter`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/serializerexporter) around the shared `serializer.MetricSerializer`, translating OTLP metrics with the same [`pkg/opentelemetry-mapping-go/otlp/metrics`](<<<SRC>>>/pkg/opentelemetry-mapping-go/otlp/metrics) translator used by OTLP ingest, and joining the classic pipeline at [serialization](../pipelines/metrics/serialization.md). APM stats emitted by the `datadog` connector arrive on this path encoded as OTLP metrics; `consumeStatsPayload` decodes them back into `StatsPayload` values and re-injects them into the trace-agent. Retry and queue defaults are reconciled with the legacy forwarder budget in [`exporter.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/serializerexporter/exporter.go) (2sâ†’64s backoff, 15-minute budget).
1. **Logs**: wraps [`logsagentexporter`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/logsagentexporter) with the `otel_source:otel_agent` tag, feeding the logs-agent pipeline running inside the otel-agent process; a `k8sobjects`-scoped path routes Kubernetes object logs to the [orchestrator](../containers/orchestrator.md) intake instead ([`orchestrator_exporter.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/logsagentexporter/orchestrator_exporter.go)).
1. **Host metadata**: an `inframetadata.Reporter` ([`pkg/opentelemetry-mapping-go/inframetadata`](<<<SRC>>>/pkg/opentelemetry-mapping-go/inframetadata)) consumes resource attributes from payloads marked `datadog.host.use_as_metadata` and pushes metadata via the serializer ([`host_metadata_pusher.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/serializerexporter/host_metadata_pusher.go)). Unlike the OSS exporter, DDOT never runs a periodic host-metadata loop, and the `hostname`/`only_metadata` exporter settings are ignored (warned about at factory time).

By default metrics ride the asynchronous `DefaultForwarder`, which means send failures do not propagate back into the collector's retry machinery. The alpha feature gate `datadog.serializerexporter.UseSyncForwarder` ([`featuregate.go`](<<<SRC>>>/comp/otelcol/otlp/components/exporter/serializerexporter/featuregate.go)) swaps in [`OTelSyncForwarder`](<<<SRC>>>/comp/forwarder/defaultforwarder/impl/otel_sync_forwarder.go) so failures surface through `ConsumeMetrics` and OTel's queue/retry takes over; HTTP 400/413/403 are mapped to permanent (non-retryable) errors.

## The infraattributes processor and the datadog connector

The [`infraattributesprocessor`](<<<SRC>>>/comp/otelcol/otlp/components/processor/infraattributesprocessor) resolves entities named by resource attributes â€” `container.id`, `k8s.pod.uid`, `k8s.deployment.name` plus namespace, `aws.ecs.task.arn`, `oci.manifest.digest`, `process.pid`, node and namespace metadata â€” against the tagger (remote in connected mode, local in standalone), then merges entity tags and global tags back into resource attributes, mapping `service`/`env`/`version` onto the OTel unified-service-tagging conventions. When `container.id` is missing it can generate one from origin info (PID, cgroup inode, or pod UID + container name). `cardinality: 0|1|2` selects tagger cardinality (low/orchestrator/high). `trace_container_tag_promotion: off|duplicate|rename` â€” traces pipelines only â€” controls prefixing custom tags with `datadog.container.tag.` so the trace-agent promotes them into `_dd.tags.container`.

The [`datadogconnector`](<<<SRC>>>/comp/otelcol/otlp/components/connector/datadogconnector) runs the trace-agent's `stats.Concentrator` and obfuscator over spans flowing through OTel pipelines and emits APM stats as OTLP metrics, which the `datadog` exporter's metrics path converts back into `StatsPayload` for the trace-agent. This decoupling is what enables "sampled traces, complete stats": route *unsampled* spans to `datadog/connector` for stats in one pipeline, and sample (for example with `tailsampling`) before the `datadog` exporter in another. Because the traces exporter always declares client-computed stats, **a pipeline without the connector produces no APM stats at all** â€” this is why the default config always includes it.

## Diagnostic extensions

1. **`ddflare`** ([`impl/extension.go`](<<<SRC>>>/comp/otelcol/ddflareextension/impl/extension.go)): an HTTPS server (default `:7777`, `otelcollector.extension_url`) serving the provided/enhanced configs, the env table, build info (including the BYOC flag), and a list of debug-source URLs (pprof, zpages, health_check, prometheus). When DDOT runs alongside a core agent, the server uses the Agent IPC TLS material and auth-token middleware; OCB-embedded builds fall back to a self-signed certificate.
1. **`ddprofiling`** ([`impl/extension.go`](<<<SRC>>>/comp/otelcol/ddprofilingextension/impl/extension.go)): runs the dd-trace-go profiler on the DDOT process itself. In agent mode profiles are shipped through the embedded trace-agent's profiling proxy (local server, default port `7501`); in standalone mode they go directly to the intake.
1. **`dogtel`** ([`impl/extension.go`](<<<SRC>>>/comp/otelcol/dogtelextension/impl/extension.go), standalone mode only): host metadata collection, a tagger gRPC server for other local consumers, and secret-backend settings read from the OTel config.

The core agent integrates with these: its flare filler ([`flare_filler.go`](<<<SRC>>>/comp/otelcol/collector/impl-pipeline/flare_filler.go)) calls the ddflare extension over the authenticated IPC HTTP client, stores customer vs runtime configs under `otel/otel-flare/` in the [flare](../operations/flare.md), and scrapes each advertised debug source. The "OTel Agent" section of `agent status` ([`comp/otelcol/status/impl/status.go`](<<<SRC>>>/comp/otelcol/status/impl/status.go)) combines the ddflare response with the collector's prometheus self-metrics (receiver accepted/refused, exporter sent/failed). The DDOT binary also has its own `otel-agent status` and `otel-agent flare` subcommands.

## Component set, OCB generation, and BYOC

The upstream components compiled into DDOT are declared in the OCB (OpenTelemetry Collector Builder) manifest [`comp/otelcol/collector-contrib/impl/manifest.yaml`](<<<SRC>>>/comp/otelcol/collector-contrib/impl/manifest.yaml), currently pinned to collector v0.155.0. It includes, among others: receivers `otlp`, `prometheus`, `filelog`, `hostmetrics`, `jaeger`, `zipkin`, `kubeletstats`, `k8sobjects`, `fluentforward`, `dockerstats`, `receivercreator` with the observer extensions; processors `batch`, `memory_limiter`, `transform`, `filter`, `k8sattributes`, `resourcedetection`, `tail_sampling`, `probabilistic_sampler`; exporters `otlp`, `otlphttp`, `loadbalancing`, `debug`, `nop`; extensions `health_check`, `pprof`, `zpages`, `file_storage`, and the upstream `datadog` extension. `dda inv collector.generate` ([`tasks/collector.py`](<<<SRC>>>/tasks/collector.py)) downloads the matching OCB release and regenerates [`components.go`](<<<SRC>>>/comp/otelcol/collector-contrib/impl/components.go) with `--skip-compilation`; the upstream `datadog` exporter and connector are stripped from the generated output because the Agent registers its own flavors.

/// warning
The collector version is duplicated: the manifest pins v0.155.0 and [`collector/impl/collector.go`](<<<SRC>>>/comp/otelcol/collector/impl/collector.go) hardcodes the same string in the collector `BuildInfo`. Both must be bumped together when upgrading (the `update-otel-deps` skill in this repository automates the whole dance).
///

**BYOC (bring your own components)** lets customers compile additional upstream components into DDOT. The workflow is: edit the manifest, run `dda inv collector.generate`, then `dda inv otel-agent.build --byoc` ([`tasks/otel_agent.py`](<<<SRC>>>/tasks/otel_agent.py)), which sets the `cmd/otel-agent/command.BYOC` ldflag so the custom build is identifiable in ddflare responses and telemetry. Two distribution paths exist:

1. **Kubernetes**: [`Dockerfiles/agent-ddot/Dockerfile.agent-otel`](<<<SRC>>>/Dockerfiles/agent-ddot/Dockerfile.agent-otel) clones the repository at the matching Agent tag, injects the custom manifest, builds, and layers the resulting binary onto the published Agent image.
1. **Linux/Windows hosts**: [`tools/build-ddot-byoc`](<<<SRC>>>/tools/build-ddot-byoc) replaces the `otel-agent` binary inside the Agent OCI package's `ddot` extension layer and pushes the result to a private registry, from which `datadog-agent otel install --registry ...` ([`cmd/agent/subcommands/otel/command.go`](<<<SRC>>>/cmd/agent/subcommands/otel/command.go)) installs it through the [fleet installer](../deployment/fleet.md) extension mechanism.

## Deployment modes

1. **Kubernetes DaemonSet (the primary target)**: DDOT runs as an extra container in the Agent DaemonSet pod; the Helm chart and Datadog Operator provide the container and a default config. Connected mode applies â€” remote tagger, hostname, and config sync all reach the core-agent container over pod-local IPC. [`Dockerfiles/agent-ddot/Dockerfile`](<<<SRC>>>/Dockerfiles/agent-ddot/Dockerfile) builds the [image variant](../deployment/container-images.md) with the binary at `/opt/datadog-agent/embedded/bin/otel-agent`.
1. **Gateway**: a standalone collector Deployment behind a Service, receiving OTLP from daemonset collectors or SDKs. `otelcollector.gateway.mode: true` flips the `datadog` extension's `deployment_type` and enables gateway usage telemetry; there is no core agent, so no remote tagger or config sync.
1. **Linux hosts**: shipped as the `datadog-agent-ddot` deb/rpm ([`omnibus/config/software/datadog-otel-agent.rb`](<<<SRC>>>/omnibus/config/software/datadog-otel-agent.rb), Bazel migration in [`packages/ddot`](<<<SRC>>>/packages/ddot)) or as the fleet-installer `ddot` OCI extension, whose binary lands under `/opt/datadog-packages/datadog-agent/<version>/extensions/ddot/embedded/bin/otel-agent`. See [Packaging](../deployment/packaging.md).
1. **Windows**: `otel-agent.exe` runs as the `datadog-otel-agent` service ([`cmd/otel-agent/windows/controlsvc`](<<<SRC>>>/cmd/otel-agent/windows/controlsvc)).
1. **Standalone** (`otel_standalone: true`): no core Agent on the host at all; local tagger, workloadmeta, hostname, and secrets, with the `dogtel` extension supplying host metadata and a tagger server.
1. **ECS/Fargate**: DDOT runs as a container in the task, connected to the agent container; infraattributes resolves `aws.ecs.task.arn` entities.

| dogtel tagger gRPC | otel-agent (standalone) | Serves tagger data to other local consumers |

Notably absent: port 8126. The embedded trace-agent's HTTP receiver is forced off, so DDOT does not accept Datadog-native tracer traffic â€” that remains the job of the regular trace-agent process.

## Gotchas

1. **Two collectors, one directory.** [`comp/otelcol/collector/impl-pipeline`](<<<SRC>>>/comp/otelcol/collector/impl-pipeline/pipeline.go) is OTLP ingest inside the core agent; [`comp/otelcol/collector/impl`](<<<SRC>>>/comp/otelcol/collector/impl/collector.go) is DDOT. The Fx modules are `fx-pipeline` vs `fx` respectively.
1. **The converter mutates config invisibly.** The running ("enhanced") config differs from what the user wrote; `<component>/dd-autoconfigured` names in a running config are converter artifacts. Debug against the flare's `otel/otel-flare/runtime.cfg`, not the user's YAML.
1. **A `datadog` exporter is structurally required.** Without one, DDOT boots a bare collector with no serializer, no trace-agent, and no logs agent; with more than one, config bridging fails.
1. **No connector, no APM stats.** The traces path always sets `Datadog-Client-Computed-Stats: true`, so removing `datadog/connector` from the pipeline silently produces zero APM stats.
1. **Metrics send failures are silent by default.** The async forwarder swallows delivery errors; enable the alpha `datadog.serializerexporter.UseSyncForwarder` feature gate to propagate them into OTel retry. Retry semantics differ subtly between the two paths.
1. **zlib is load-bearing.** DDOT forces zlib metrics compression and disables the v3 series API in two places; changing one without the other breaks metric intake.
1. **Secret and API key resolution order surprises.** In connected mode the converter replaces an `ENC[...]` API key in collector YAML with the core agent's resolved value; without `--core-config`, `LoadDatadog` never runs and proxy settings are only honored through an explicit `LoadProxyFromEnv` call.
1. **All-digit API keys** would unmarshal as integers and fail confmap unmarshaling; `apiKeyItoa` in the config bridging works around this.
1. **`hostname` and `only_metadata` in the exporter config are ignored** â€” host metadata is push-only from resource attributes, never from source providers; warnings are logged at factory time.
1. **`hostmetrics` in connected mode double-reports** system metrics already collected by the core agent; the converter warns but does not block.
1. **Container tag promotion is traces-only.** `trace_container_tag_promotion` never affects metrics, logs, or profiles.
