# pkg/proto

## Purpose

`pkg/proto` is the canonical home for all inter-component communication contracts used by the
Datadog Agent. It contains:

1. **`.proto` source files** under `pkg/proto/datadog/` — the authoritative schema definitions
   for every gRPC service and every shared message type used across agent binaries.
2. **`pkg/proto/pbgo/`** — Go code generated from those schemas (protobuf, vtprotobuf, and msgpack
   serialisers). This sub-package is pure generated code and is documented separately in
   `SKIPPED.md`.
3. **`pkg/proto/msgpgo/`** — A small hand-written package that defines `RemoteConfigKey`, a struct
   that is serialised with MessagePack (via `tinylib/msgp`) and embedded in Remote Configuration
   API keys.
4. **`pkg/proto/utils/`** — A reflection-based helper (`ProtoCopier`) for shallow-copying protobuf
   message values without importing generated code directly.

`pkg/proto` lives in its own Go module (`go.mod` at `pkg/proto/`) so that the generated code can
be versioned and vendored independently of the rest of the agent.

Code generation is driven by `dda inv protobuf.generate`. External proto dependencies (Google
well-known types, `go-proto-validators`) are pinned in `protodep.toml` and checked in under
`pkg/proto/protodep/`.

### `pkg/proto/msgpgo` — RemoteConfigKey

```go
type RemoteConfigKey struct {
    AppKey     string `msgpack:"key"`
    OrgID      int64  `msgpack:"org"`
    Datacenter string `msgpack:"dc"`
}
```

`RemoteConfigKey` represents the structured content of a Remote Configuration API key. The key is
transmitted as a base32-encoded, MessagePack-serialised blob with the prefix `DDRCM_`. The
generated `key_gen.go` file provides `MarshalMsg` / `UnmarshalMsg` / `EncodeMsg` / `DecodeMsg`
methods (produced by `tinylib/msgp`), making zero-copy decoding possible.

### `pkg/proto/utils` — ProtoCopier

```go
func ProtoCopier(v interface{}) func(v interface{}) interface{}
```

`ProtoCopier` uses reflection at initialisation time to discover all public fields of a protobuf
struct that have a matching `Get<FieldName>()` accessor (the pattern generated by `protoc-gen-go`).
It returns a reusable copying function that shallow-copies a struct of that type without triggering
a full `proto.Clone`. Useful when the caller needs a snapshot of a proto message but does not want
the overhead of a deep copy.

### Code generation and patches

The `patches/` directory contains two patches applied on top of the generated
`pbgo/trace/span_gen.go` file:

- **`0001-Customize-msgpack-parsing.patch`** — Adds nil-safe `parseStringBytes` wrappers around
  string field deserialisers so that null values in MessagePack payloads from older tracer versions
  are tolerated and decoded as empty strings rather than causing an error.
- **`0002-Make-nil-map-deserialization-retrocompatible.patch`** — Makes nil-map deserialisation
  backwards compatible for `Span.Meta` and `Span.Metrics`.

These patches exist because `tinylib/msgp` generates strict deserializers; the patches widen
acceptance to handle wire formats emitted by older tracer SDKs.

## Usage

### gRPC clients and servers

Generated service interfaces live in `pkg/proto/pbgo/core` (aliased as `pbgo`). The standard
pattern is:

```go
import pbgo "github.com/DataDog/datadog-agent/pkg/proto/pbgo/core"

// Server side — implement the generated interface
type agentServer struct{}
func (s *agentServer) GetHostname(ctx context.Context, _ *pbgo.HostnameRequest) (*pbgo.HostnameReply, error) { ... }

// Client side — dial and call
conn, _ := grpc.Dial(addr, opts...)
client := pbgo.NewAgentClient(conn)
reply, _ := client.GetHostname(ctx, &pbgo.HostnameRequest{})
```

### Remote Configuration key parsing (`pkg/config/remote/service`)

`pkg/config/remote/service` decodes RC keys in authentication headers using `msgpgo.RemoteConfigKey`:

```go
import "github.com/DataDog/datadog-agent/pkg/proto/msgpgo"

rawKey, _ := base32.StdEncoding.WithPadding(base32.NoPadding).DecodeString(strings.TrimPrefix(rcKey, "DDRCM_"))
var key msgpgo.RemoteConfigKey
key.UnmarshalMsg(rawKey)
// key.AppKey, key.OrgID, key.Datacenter are now populated
```

### APM trace pipeline

The APM trace-agent and the trace package inside the core agent rely heavily on
`pkg/proto/pbgo/trace` types (`Span`, `TraceChunk`, `TracerPayload`, `AgentPayload`, `StatsPayload`).
The `pkg/trace/semantics` and `pkg/trace/stats` packages import `pbgo/trace` types directly for
span processing and stats computation.

### Security module

`pkg/security/module` uses the `AgentSecure` gRPC service (defined in `api.proto`) to stream
security events and register with the core agent.

## Regenerating code

```bash
dda inv install-tools        # ensure protoc, protoc-gen-go, vtprotobuf are in PATH
dda inv protobuf.generate    # regenerates pbgo/ and re-applies patches/
```

After regeneration, re-apply `patches/` manually if the base generated file changes in a way that
causes patch conflicts.

## Cross-references

### How pkg/proto fits into the wider system

```
Datadog RC backend
      │  HTTPS POST /api/v0.1/configurations
      │  body: pbgo.LatestConfigsRequest (remoteconfig.proto)
      ▼
pkg/config/remote/api.HTTPClient → pkg/config/remote/service.CoreAgentService
      │  serves gRPC over IPC (api.proto: AgentSecure.ClientGetConfigs)
      ▼
pkg/config/remote/client.Client (trace-agent, system-probe, security-agent)

APM SDKs (msgpack/JSON)
      ▼
pkg/trace/api.HTTPReceiver → decoded into pb.TracerPayload (tracer_payload.proto)
      │  processed through pkg/trace/agent pipeline
      ▼
pkg/trace/writer.TraceWriter → pb.AgentPayload (agent_payload.proto) → Datadog backend

pkg/security/module
      │  connects to AgentSecure gRPC server (api.proto) via UDS
      └─ streams security events, registers for configuration
```
