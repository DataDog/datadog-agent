# Integration labs (agint:generate-lab)

This directory holds Datadog Agent E2E framework labs generated by the
`agint:generate-lab` skill — one subdirectory per integration. They are owned by
`@DataDog/agent-integrations` and kept separate from the framework's default
environment-creation scenarios.

## Task surface

Every lab exposes the same commands under the `aws.integrations.<integration>`
namespace (task modules live in `tasks/e2e_framework/aws/integrations/`):

```bash
dda inv aws.integrations.<integration>.create
dda inv aws.integrations.<integration>.status
dda inv aws.integrations.<integration>.check
dda inv aws.integrations.<integration>.reload-check   # existing-check VM labs only
dda inv aws.integrations.<integration>.exec --role <role> --command '<cmd>'
dda inv aws.integrations.<integration>.ssh --role <role>
dda inv aws.integrations.<integration>.destroy
```

- **create** deploys the scenario and installs/configures the Datadog Agent and check.
- **status** runs `datadog-agent status` on the Agent host.
- **check** runs the lab's Agent check.
- **reload-check** reinstalls the check from its source ref and re-runs configcheck/check.
- **exec / ssh** run a command on / open SSH to an owned host role.
- **destroy** tears down resources owned by the lab.

Kubernetes labs use the generated kubeconfig instead of `--role`: `ssh` prints
`export KUBECONFIG=<path>` and `exec` runs `kubectl` locally.

The shared task implementation lives in
`tasks/e2e_framework/aws/integrations/_lab_tasks.py` (`build_collection` for VM labs,
`build_kube_collection` for Kubernetes labs); each lab's module only declares its
constants.

## Deployment (deploy=true labs)

For labs deployed on live infrastructure, `create` installs and configures the Datadog
Agent on the Agent host. The `exec`/`status`/`check`/`ssh` tasks resolve hosts from the
scenario's Pulumi stack outputs (by role), not a separate metadata file.
Destroy ownership: `destroy` removes only resources the lab created; it never mutates
user-provided or external systems.

## What each lab deploys

Per-lab specifics (topology, workload, and the check it exercises) are documented in
each scenario's `run.go` package comment, e.g. `redisdb/run.go`.

## Adding a lab

A lab is normally generated by `agint:generate-lab`. It consists of:

- a scenario package `<integration>/`, registered in `registry.go`
  (key `aws/integrations/<integration>`);
- a task module `tasks/e2e_framework/aws/integrations/<integration>.py` that calls
  `build_collection(...)` / `build_kube_collection(...)`, registered in that package's
  `__init__.py`.

The top-level `registry/scenarios.go` and `tasks/e2e_framework/aws/__init__.py`
already include the whole `integrations` group, so new labs do not edit them.
