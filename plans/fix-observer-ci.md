## Category 1 — Component framework violations

**Failing check:** `lint_components`

**Root cause (confirmed via `dda inv components.lint-components`):**

The logssource component added by PR #49481 has three violations:

1. `comp/observer/logssource/impl/component.go` is named `component.go`.  
   The `lint_components` task scans `comp/**` and detects any directory containing a
   `component.go` as a v1 component definition. The `impl/` directory is incorrectly
   treated as a standalone v1 component because it has that file, which:
   - expects a `Component` interface (not present — it's an impl)
   - expects a `// team:` annotation (not present in the impl file)

2. `comp/observer/logssource/impl/component.go` imports `go.uber.org/fx` directly.  
   Impl files must not import `fx` — all lifecycle wiring must use `compdef.Lifecycle`
   / `compdef.Hook` (same fix applied to `comp/observer/impl/observer.go` in #48895).

3. `comp/README.md` and `.github/CODEOWNERS` are not updated for the new component.  
   These are auto-generated from the component tree; they differ from what
   `lint_components` would generate.

**Fixes:**

- [x] Rename `comp/observer/logssource/impl/component.go` → `comp/observer/logssource/impl/logssource.go`
  (eliminates the spurious v1 component detection)

- [x] In `comp/observer/logssource/impl/logssource.go` (after rename), replace `fx.Lifecycle`
  and `fx.Hook` with `compdef.Lifecycle` and `compdef.Hook`, and drop the `go.uber.org/fx`
  import. Pattern from `comp/observer/impl/observer.go`:
  ```go
  Lc compdef.Lifecycle
  // ...
  deps.Lc.Append(compdef.Hook{OnStart: ..., OnStop: ...})
  ```

- [x] Run `dda inv components.lint-components --fix` to regenerate `comp/README.md` and
  `.github/CODEOWNERS`.

## Category 3 — Stale go.mod dependencies

**Failing checks:** `datadog_otel_components_ocb_build`, `integration_tests_otel`

**Root cause (confirmed by inspection):**

The logssource PR (#49481) removed `comp/observer/def` from the Go code in
`pkg/logs/pipeline` and `pkg/logs/processor`, but did not run `go mod tidy`.
Both `go.mod` files still declare the dependency:

```
# pkg/logs/pipeline/go.mod and pkg/logs/processor/go.mod
github.com/DataDog/datadog-agent/comp/observer/def v0.0.0-00010101000000-000000000000
```

No `.go` file in those packages actually imports `comp/observer/def` any more.
The OTel build checks (`go mod tidy`-based) fail because of this stale entry.

**Fix:**

- [x] `cd pkg/logs/pipeline && go mod tidy`
- [x] `cd pkg/logs/processor && go mod tidy`
- [x] Verify no other packages gained/lost `comp/observer/def` as a dep after the logssource change.

## Category 4 — Protobuf version mismatch

**Failing check:** `protobuf_test`

**Root cause (confirmed):**

`.protoc-version` contains `34.0` (bumped by a main merge), but the committed
generated files `pkg/proto/pbgo/core/observer.pb.go` and `observer_grpc.pb.go` were
generated with `protoc v5.29.3` (i.e. libprotoc 29.3):

```
// Code generated by protoc-gen-go. DO NOT EDIT.
// versions:
//  protoc-gen-go v1.36.11
//  protoc        v5.29.3
```

The CI job runs `dda inv protobuf.generate --pre-commit`, which:
1. Checks `protoc --version` matches `.protoc-version` (34.0 ≠ 29.3 → `Exit`)

**Fix:**

- [x] Install protoc 34.0 locally (`dda inv install-protoc`)
- [x] Run `dda inv protobuf.generate` to regenerate all proto files with protoc 34.0
- [x] Commit the updated `.pb.go` files

> Note: if protoc 34 changes the generated output, the diff may be large. Only
> `pkg/proto/pbgo/core/observer.pb.go` and `observer_grpc.pb.go` are observer-specific;
> the regeneration task may touch others too.

## Category 5 — Static quality gates

**Failing check:** `static_quality_gates`

**Root cause (suspected, no log access):**

The job builds agent packages (deb, rpm, etc.) and compares actual on-disk / on-wire
sizes against thresholds in `test/static/static_quality_gates.yml`. The thresholds in
the file were last updated when main was merged into q-branch-observer (commit #48883).

Since then, observer + logssource code has been added to the agent binary (both are
included via `observer.Bundle()` in `cmd/agent/subcommands/run/command.go`). The
additional binary size likely exceeds the current thresholds.

**Fix:**

- [ ] Build the agent deb/rpm packages locally or in CI to get actual sizes:
  `dda inv agent.build --build-exclude=systemd && dda inv agent.omnibus-build`
- [ ] Update thresholds in `test/static/static_quality_gates.yml` to match actual sizes
  (round up by ~5 MiB headroom)
- [ ] Follow the size-exception Confluence process if the bump exceeds the approval
  threshold (typically >2% increase requires a Jira exception ticket)

> This fix may be blocked if the static QG process requires a separate exception ticket.
> In that case, create the ticket and add a reference in the PR description.
