### Key Allocation Hotspots (Non-Test)

1. **`pkg/metrics.(*Gauge).flush`** — #1 hotspot. Allocates a `*Serie` struct and a `[]metrics.Point` slice for every gauge context on every flush. At 1000 contexts × 15s flush interval = 1000 Serie allocations/flush. **Target for US-004 (SeriePool).**

2. **`pkg/aggregator.(*contextResolver).trackContext`** — #4 hotspot. Allocates:
   - A `*Context` struct for each new (metric, host, tags) combination seen
   - Tag buffer (`HashingTagsAccumulator`) for key generation per call
   **Target for US-003 (HashingTagsAccumulator pooling).**

3. **`pkg/metrics.ContextMetrics.AddSample`** — #5 hotspot. The `map[ckey.ContextKey]Metric` is freshly allocated per time bucket (`MakeContextMetrics()`). At high flush rates, this is significant. **Target for US-004 (ContextMetrics reuse).**

4. **`pkg/aggregator/internal/tags.(*Store).Insert`** — #6 hotspot. Tag store insertion creates tagged entries per sample. Related to tag slice allocation patterns. **Partially addressed by US-003.**

5. **`pkg/tagset.hashedTags.Copy`** — #7 hotspot. Tag slice copy during context key generation. Copying tags into sorting buffer per trackContext() call. **Target for US-003 (in-place sorting).**

### Key CPU Hotspots (Non-GC, Non-Test)

1. **GC overhead (ranks 1, 4, 5, 6)** — Combined ~22% of CPU time is pure GC work (`scanobject`, `findObject`, `mspan.base`, `gcBits.bitp`). This is the cost of the current allocation rate — **reducing allocations will directly cut GC time.**

2. **`sum128internal` / context key hashing** — #2 hotspot at 8.34% flat. The 128-bit hash computation is called for every `trackContext()` invocation. Non-trivially reducible — the hash is inherently O(n_tags).

3. **`pkg/tagset.(*HashGenerator).Dedup2` + `sort.partition_func`** — Combined ~5% flat CPU for tag dedup and sorting before hashing. Sorting tags into canonical order is required for key stability but creates a copy. **Target for US-003 (in-place sort).**

4. **`pkg/aggregator.(*contextResolver).trackContext`** — 26.58% cumulative CPU — the dominant call chain. Includes tag buffer operations, context key generation, and context map updates.

## Key File Analysis (from static code review)

### `comp/dogstatsd/server/server_worker.go`
- `MetricSampleBatch` (`samples` slice, line 35) is already reused per packet via reset to length 0 — **good pattern**
- No major per-sample allocations visible

### `pkg/aggregator/context_resolver.go`
- `trackContext()` (line 104+): `taggerBuffer` and `metricBuffer` are per-resolver fields, reused across calls
- `*Context` struct is allocated once per new (metric_name, host, tags) combination — not pooled
- Context population loop builds composite tags (line 126-127)

### `pkg/aggregator/time_sampler.go`
- `MakeContextMetrics()` (line 100, 152): allocates a fresh `map[ckey.ContextKey]Metric` per time bucket
- `serieBySignature` map (line 159): reallocated on every `flushSeries()` call
- `countersSampleZeroValue()` (line 287): allocates a temporary `*MetricSample` per expired counter

### `pkg/aggregator/sketch_map.go`
- Inner map `make(map[ckey.ContextKey]*quantile.Agent)` (line 56): allocated per new timestamp bucket
- No pre-sizing hint — may rehash as cardinality grows

### `pkg/metrics/series.go`
- `Serie.flush()` creates `*Serie` and `[]Point` slice per context per flush — highest alloc site

### `comp/forwarder/defaultforwarder/transaction/transaction.go`
- `NewHTTPTransaction()` (line ~286): allocates `&HTTPTransaction{}` + `make(http.Header)` per transaction
- At 15s flush intervals with 1-N domains × payloads, this is moderate frequency
- Baseline: **784 B/op, 15 allocs/op per single-payload transaction**

### `comp/forwarder/defaultforwarder/default_forwarder.go`
- `createAdvancedHTTPTransactions()`: allocates transaction slice per submission
- Already pre-sized with `make([]*transaction.HTTPTransaction, 0, capacity)` — good pattern

## Notes for Future Comparison

- All baselines measured with `count=3, benchtime=500ms` on branch `ralph/dogstatsd-to-forwarder-optimisations` at commit off main
- For US-008 final validation, use `count=5` and `benchstat` for rigorous comparison
- GC overhead (ranks 1, 4, 5, 6 in CPU profile) represents ~22% of total CPU — reducing allocs will automatically reduce this
- The `BenchmarkTimeSamplerSampleTagCount` reports 0 allocs/op because the context is warm (pre-populated) — allocations happen at context creation time, captured in `trackContext`
- Histogram type (22,040 allocs/op) is exceptionally high because each histogram flushes 7 different percentile series (`count`, `sum`, `min`, `max`, `50p`, `95p`, `99p`) — each is a separate `Serie` + `[]Point` allocation
