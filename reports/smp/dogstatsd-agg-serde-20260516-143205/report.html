DogStatsD hack-week experiment report

Can DogStatsD become a streaming data system?

This report summarizes a local SMP experiment series exploring whether DogStatsD can grow new observability features—debug stats, capture/replay, future lookback queries, and cheaper payload construction—without rebuilding the same parsing, identity, and aggregation work in parallel.

Why we ran this experiment

Adding DogStatsD features is hard because DogStatsD is on a very hot path. New observability features often need the same ingredients as aggregation—parsed metric names, tags, hosts, origin information, context keys, recent values, and sometimes raw bytes. If each feature builds its own copy of that state, the Agent pays the parsing, hashing, allocation, and memory cost multiple times.

Our immediate motivation was a future lookback/query capability: answer local questions like “what DogStatsD series were active in the last 30 seconds?” or “what changed at a 1s window?” without building a second mini-aggregator beside the real one. More broadly, this is the same pressure behind efforts like comp/observer: useful local observability features are natural to build as a standalone observer, but the add-on approach repeats work and struggles to fit the Agent's resource envelope.

The experiment asked: what if we applied those observer/database ideas to the core DogStatsD path instead, so the core path gains headroom and new features become cheap views over shared data?

30-second primer: what changes in the hot path?

Today: each line becomes fresh objects

A common DogStatsD metric line might look like:

requests.count:1|c|#env:prod,service:web,region:us1

The normal path parses that into strings, tag slices, host/origin state, a sample object, shard/context keys, aggregate state, and finally payload rows. That is correct and general, but repeated traffic rebuilds a lot of the same description.

Experiment: keep the description, stream the value

When the metric name and tag list repeat, we can remember the parsed description and pass a small reference through the pipeline. The hot path then mostly handles the changing parts: value, sample rate, timestamp/window, and aggregate update.

Compatibility still comes from fallback: unusual metric types, origin/enrichment cases, timestamps, events, service checks, and unknown tagsets continue through the general path unless explicitly tested.

Validated direction

Stop rebuilding the same metric description

-14% agent CPU

In today's general path, every DogStatsD line is parsed into names, tag strings, tag slices, and a MetricSample-like object even when it is the same metric and tags as the previous line. High-rate repeated-tags win

+28.4% throughput
Scope: high-rate UDS metrics-only workload versus main; not a mostly-unique tag workload.

DogStatsD traffic often repeats the same comma-separated tag list many times. Caching the parsed form of those repeated tag lists produced a large win in that workload. This remains gated until adversarial/mostly-unique traffic is tested.

UDS bounded-ingress win

+7.4% high-rate
Scope: UDS datagram high-rate case; UDP, stream sockets, and origin metadata need more work.

The Agent has to absorb bursts between socket readers and parser workers. A compact byte-budgeted UDS ring plus batched parser drains was the best bounded-ingress design tested, replacing large hidden packet-channel backlog with explicit backpressure.

Mixed result

Memory is predictable, not lower yet

+11 MiB live heap
Scope: Stage S late heap profile; RSS deltas were sometimes larger.

Heap profiles showed the mature design was only about +11 MiB live heap versus main, and the raw ring used less explicit ingress-reservoir heap than the packet pool. RSS was still higher in some runs because the prototype retains dictionaries/caches; this needs byte budgets before any default-on claim.

How we feel about the experiment: it moved from a sketch to a validated direction. We found real macro wins, falsified several attractive shortcuts, and learned that small IDs for repeated metric descriptions only pay off when they let us avoid building the old object-heavy representation in the first place. We do not yet know the final throughput ceiling because the latest CPU-winning runs are fixed-rate and not saturated.
Claim boundaries: the strongest numbers are local SMP results on optimized Linux/arm64 images, mostly UDS datagram traffic, often metrics-only and repeated-tag workloads. Most new paths are behind explicit experimental gates. Treat the results as evidence for architecture direction and headroom, not as a ready-to-enable production claim across UDP, origin detection, Windows/named pipes, all metric types, events, service checks, or mostly-unique tagsets.

Motivation: avoid building a second DogStatsD inside DogStatsD

The normal Agent path is intentionally optimized for production aggregation: listeners read packets, the parser turns each line into metric samples with strings and tag slices, DogStatsD chooses an aggregation pipeline, the aggregator computes backend context keys, and the serializer later builds payload-shaped data.

That design is general and battle-tested, but it makes new local features expensive when they are layered on from the side. A feature that wants to show recent activity, capture traffic around a spike, replay traffic deterministically, or expose richer dogstatsd-stats usually needs to rediscover some of the same facts the main path already computed.

  • Lookback/query is the motivating new feature: we want recent local views at different windows or aggregation levels without running a whole second aggregator.
  • serverDebug / agent dogstatsd-stats already exists, but historically used its own hot-path map/key computation instead of being a bounded view over shared identity state.
  • Capture/replay wants raw bytes and enough metadata to explain where they came from, but listener-specific hooks make this uneven and hard to compose with semantic views.
  • Payload construction wants dictionaries of names, tags, tagsets, resources, and values; the metrics v3 serializer has to rebuild many of these concepts downstream.
  • comp/observer-style features showed the product value of local observation, but rebuilding a tiny Agent architecture as an add-on repeats parsing/enrichment/aggregation work and makes resource budgeting difficult.

The hypothesis was that we can make these features cheaper and more consistent by treating DogStatsD like a streaming data system: keep a bounded raw ingress stream, turn parsed traffic into reusable metric facts once, and build bounded materialized views for debug, lookback, replay, and payload construction.

Plain-English guide to the techniques

The detailed notes use stage names and shorthand because the work happened incrementally. Think of the stage letters as experiment labels, not product names. This section translates the important ideas into Agent concepts.

Rule of thumb: most wins came from projection laziness. Do not build a string, tag slice, debug key, descriptor, or payload dictionary entry unless some active consumer actually needs it. When traffic repeats, prefer stable references over rebuilding the same description.

Architecture vision: raw traffic first, reusable metric facts second

In ordinary Agent terms: keep a bounded record of the bytes we received, parse those bytes into one reusable description of “what metric series is this?”, and let aggregation/debug/lookback/payload code consume that shared description instead of each rebuilding its own.

The important idea is not “columnar everything.” It is “make every expensive conversion explicit and reusable.” Raw capture/replay should consume raw envelopes; aggregation/debug/lookback should consume parsed metric facts and descriptors. DogStatsD v3 payloads are built after aggregation, ideally reusing the same descriptor/dictionary work instead of rebuilding it.

Before / after snapshots

Input queue before/after: the same idea applies at the socket boundary. Instead of relying on a large packet channel whose retained bytes are hard to reason about, the raw UDS ring gives the Agent a byte budget, oldest-age/lag telemetry, and explicit UDS backpressure when the parser falls behind.

Experimental design

We used local SMP comparisons between optimized Linux/arm64 Agent images. Each stage had a baseline image and a comparison image. SMP ran lading generators, a blackhole API endpoint, the Agent under test, and Prometheus-style metric capture. The top-line SMP goal was usually ingress throughput; explanatory metrics were queried from parquet with DuckDB.

  • Standard case: UDS datagram DogStatsD at roughly 100 MiB/s with metrics/events/service-checks mixed 90/5/5.
  • High-rate case: UDS datagram DogStatsD at 250 MiB/s, metrics only, compact batch, designed to expose parser/ingress pressure.
  • Feature-cost cases: same workload and code, toggling one experimental gate at a time when possible.
  • Replicates: most final comparisons use 3 replicates / 270 total samples; some early exploratory stages are single-replicate and are marked accordingly in notes.

What “good” means here

Throughput wins are useful when the workload is saturated or backpressured.

CPU wins matter even when fixed-rate throughput is neutral: they mean more headroom for the same traffic.

Memory wins require both less retained heap/RSS and predictable byte budgets. We are not there yet.

Compatibility remains the gate for any production migration. Most experimental paths are explicitly opt-in and fallback-safe.

Falsified

Skipping grouping naively does not win

Two shortcuts looked plausible but lost: one skipped normal grouping, and one tried one-slot direct reservation in the input ring. Both increased downstream work or contention more than they saved.

Validated

Bounded backlog is a product win

Input-queue lag telemetry lets us distinguish “actually faster” from “accepted more hidden backlog.” This matters as much as raw throughput for production behavior.

How the broader feature goals went

The experiment was not only about throughput. A major goal was to make existing and proposed DogStatsD observability features cheap enough to leave on or compose: debug stats, raw capture/replay, a future recent-lookback query feature, semantic replay, and payload construction.

The architecture wins when shared descriptors replace duplicate work, not when every view retains its own copy.

Memory read: why CPU wins did not become memory wins

The prototype often trades repeated CPU work for retained state. Main has an accidental memory advantage: it materializes many objects transiently, then lets GC reclaim them, while the experiment keeps dictionaries, compact IDs, descriptor caches, rings, and view state alive so later stages can reuse them.

The measured story is mixed rather than catastrophic: Stage S late heap profiles showed main at 162.14 MiB in-use heap and Stage R at 173.31 MiB (+11.17 MiB). RSS deltas could be larger — one standard v3 comparison was about +45.6 MiB RSS and +21.5 MiB heap metric — which points to allocator/RSS retention and cache/backlog shape, not a single giant live object. On the positive side, the raw ring lowered the explicit ingress reservoir live heap from 32.74 MiB packet-pool backlog to 18.72 MiB raw-ring storage.

Main's advantage

Fewer parallel representations: less raw tagset cache, compact ID state, descriptor ref state, and materialized view scaffolding.

Experiment's advantage

Less repeated work and more predictable overload behavior: explicit rings, reusable descriptors, and faster parser hit paths.

What has to happen

Retained-byte caps, adaptive admission, smaller per-worker defaults, cache sharing, and downstream deletion of duplicate string/tag projections.

Current memory verdict: do not claim a total-memory win. Claim less hidden ingress backlog, better predictability, and CPU headroom. Memory can become a win only if shared identity/descriptors replace multiple downstream representations instead of sitting beside them.

What we should test next

  1. Saturation/backpressure sweep: increase rate or lower CPU until the baseline develops lag/drops, then compare max stable MiB/s, lag, drops, and CPU. This converts Stage W's CPU headroom into an ingress-ceiling answer.
  2. Feature-cost matrix: measure debug stats, lookback, raw ring/capture, and combinations of those features on the latest fast path.
  3. Memory/admission hardening: adversarial mostly-unique tagsets, retained-byte caps, adaptive admission, and no-allocation telemetry.
  4. Payload dictionary reuse: use compact descriptor/name/tagset IDs directly in v3 payload dictionaries instead of rebuilding payload-local string/tag dictionaries from scratch.
  5. Compatibility breadth: mixed metric types, UDP/raw-disabled, origin-on, events, service checks, distributions/histograms/sets, timestamps, stream sockets, and named pipes.
Current verdict: the architecture is promising and increasingly real. We have decisive evidence that remembering repeated tag lists and skipping repeated object construction can move macro CPU/throughput, and that bounded ingress gives better overload honesty. We should still treat the work as experimental until saturation, memory, feature-cost, and compatibility matrices are complete.

Stage J removes that artifact and shows the narrow columnar design can beat direct metric rows. Stage L shows the large packet queue can be replaced by explicit ingress-log backpressure. Stage M shows a raw UDS ring can eliminate packet-pool backlog. Stage N shows byte-compact retained records matter. Stage O shows batched worker drains are a real but smaller high-rate ceiling step. Stage P shows one-max-slot direct reservation is not the next ceiling. Stage Q completes direct native v3/v2 payload construction but is not a large standalone win over Stage O. Stage R makes proof telemetry and naive descriptor interning explicit costs rather than hidden defaults. Stage S shows direct v3 serializer live heap is not the main RSS culprit and removes low-risk row-copy escapes. Stage T confirms parser/string interning is a real remaining bottleneck; SLRU reset avoidance is safe and exact tagset dictionaries can substantially reduce backpressure on repeated-tagset workloads. Stage U confirms compact IDs can be carried through DogStatsD batching into columnar hints. Stage V proves compact IDs can be consumed as scalar row/descriptor inputs but remains macro-neutral while the parser still materializes first. Stage W proves the missing piece: using compact identity state before materialization can produce meaningful macro CPU wins. Stage N/O plus Stage Q/R/S/T/U/V/W remains the best tested combined design; post-telemetry honesty gates narrow the claim to UDS datagram/origin-off and make high-rate backpressure visible.
Stage E/F/G/H/I/J/L/M/N/O/P/Q/R/T/U/V/W caveat: this is intentionally unsafe/local and enabled through env gates. Stage E bypasses iterable channel traversal. Stage R changes measurement cost shape by making proof shadows optional and descriptor interning opt-in.
